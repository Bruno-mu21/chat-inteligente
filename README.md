trabalho consistiu na implementação de um chat criptográfico multithread com um protocolo definido para conexão, identificação, envio e recepção de mensagens, além da especificação de um modelo de stress test para avaliar o desempenho sob diferentes cargas. No momento da conexão e identificação, o cliente conecta-se ao servidor e recebe a chave pública do servidor; em seguida o cliente gera uma chave secreta simétrica local, a cifra com a chave pública do servidor e envia essa chave cifrada ao servidor. Após isso, o cliente envia seu nome de usuário criptografado; o servidor decripta tanto a chave secreta quanto o nome, registra o novo participante e gera uma mensagem interna do tipo “(Usuário X) conectado”. Para notificar todos os participantes, o servidor encaminha essa mensagem para cada cliente, recriptografando individualmente com a chave secreta correspondente a cada cliente, garantindo confidencialidade ponta-a-ponta dentro da sala de chat. No envio de mensagens, o cliente envia ao servidor uma mensagem já criptografada com sua chave simétrica; o servidor decripta a mensagem, anexa a origem no formato “(Usuário X): mensagem” e então redistribui para todos os clientes, novamente criptografando a cópia destinada a cada receptor com a chave secreta daquele receptor, preservando privacidade entre pares. Quanto à recepção assíncrona no cliente, cada cliente opera com listener/handler assíncrono que recebe mensagens criptografadas do servidor e as decripta localmente usando sua chave secreta, permitindo interface responsiva e processamento paralelo da interface do usuário. Arquiteturalmente, o servidor é multithreaded, usando um thread pool para atender conexões e processar mensagens de forma concorrente, com controle de sincronização adequado para evitar condições de corrida ao acessar estruturas compartilhadas (lista de clientes, filas de mensagens, logs). Para validar robustez e escalabilidade, foi definido um modelo de stress test parametrizável com as seguintes variações: tamanhos do pool de threads no servidor (10, 100, 500), número de conexões simultâneas (10, 100, 500), número de pacotes enviados por conexão (10, 100, 1000) e tamanho do pacote em bytes (100 B, 1 KB, 10 KB). Esses testes permitem medir latência média, throughput (mensagens por segundo), uso de CPU/memória, ocorrência de perda de pacotes ou falhas de conexão e comportamento sob contenção de recursos. Em termos de segurança, o protocolo combina criptografia assimétrica (para troca inicial e distribuição segura de chaves) com criptografia simétrica (para eficiência na transmissão das mensagens), garantindo confidencialidade e minimizando overhead; o servidor atua como ponto de mediação que nunca envia chaves secretas em texto claro e recriptografa mensagens individualmente para cada receptor, reduzindo riscos caso um canal seja comprometido. Por fim, o sistema deve incluir logs de eventos (conexões, desconexões, erros) e métricas geradas durante os stress tests para análise posterior, oferecendo um conjunto completo para avaliação funcional e de desempenho do chat criptográfico multithread.
